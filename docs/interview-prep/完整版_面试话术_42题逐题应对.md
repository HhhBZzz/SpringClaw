### 题目 2：项目中最难的 Agent 技术点是什么？怎么解决的？重新设计会怎么改？

**推荐回答：**

> "最难的是自主循环的虚假完成问题。模型输出'TASK_COMPLETE'标记来说任务完成了，但实际上它什么工具都没调、什么文件都没写——它学会了用完成标记来结束循环，而不是真正完成任务。在我的测试中，第一版大约 30% 的写任务会虚假完成。
>
> 解决方案是 AutonomousExecutionTracker。它用 ThreadLocal 在每次循环中跟踪真实的工具调用——记录 hasWriteToolCall（是否调用了写工具）、createdFiles（创建了哪些文件）、modifiedFiles（修改了哪些文件）。写任务的完成条件是 `hasWriteToolCall AND (createdFiles 或 modifiedFiles 非空)`，读任务可以纯文本完成。如果模型声称完成但 tracker 显示没有真实副作用，系统会注入拒绝 prompt 强制模型继续操作。这把虚假完成率降到了接近 0。
>
> 但引入了新问题——有时候工具调用抛了异常，但文件实际写入了，tracker 没记录到。所以我做了 risk-level-aware 的分级策略：read-only 任务可以纯文本完成，write 任务必须有文件变更证据，side_effect 任务必须有成功的命令执行记录。
>
> 如果重新设计，我会加 Checkpoints——每次工具调用前做文件快照（类似 Cline 的设计），这样 Agent 改坏了代码可以回滚。还会把 ThreadLocal 改成 ConcurrentHashMap 以支持未来的异步执行。"

**代码映射：** `AutonomousLoopEngine.java` lines 272-308（fake completion guard），`AutonomousExecutionTracker.java`（整个类）

## 第二章：Agent 架构

### 题目 3：你项目中多 Agent 协作的架构——Supervisor / Orchestrator / Pipeline / Swarm，为什么选这个模式？

**推荐回答：**

> "SpringClaw 不是传统意义上的多 Agent 系统。它采用的是单 Agent + 多引擎 + 轻量 Orchestrator 模式。
>
> 具体来说，Coordinator 层（也就是 AgentDecisionService + EngineSelector）接收请求后做意图分类，然后选择 6 个引擎中的一个来执行。大部分请求是单引擎直接执行。但对于复合任务——比如用户说'搜索最新 AI 新闻并写入文件'——Coordinator 会拆成 web_search + file_write 两个子任务，分别选引擎执行后合并结果。
>
> 为什么不做 Supervisor/Orchestrator-SubAgent（像 CrewAI 那样）或 Swarm？因为对于本地 Agent 场景，单 Agent 多引擎已经够用。多 Agent 引入三个成本：1) Agent 间通信延迟（每次委派都是一次模型调用）；2) 一致性问题（多个 Agent 同时修改文件怎么办）；3) 调试复杂度（链路太长，出错难定位）。SpringClaw 的设计哲学是'用最简单的架构解决问题'——如果未来需要处理更复杂的任务（比如同时分析 10 个文件并生成报告），我会加多 Agent 层，但不是现在。"

**代码映射：** `EngineSelector.java`（优先级选择），`AgentDecisionService.java`（意图分类）

**追问应对：** 如果面试官追问"那你了解 CrewAI 吗"——"了解核心概念：Agent 有 role/goal/backstory，Task 有 description/expected_output，Process 有 sequential/hierarchical。CrewAI 的 sequential 流程对应 Pipeline，hierarchical 对应 Supervisor。如果我要在 SpringClaw 中实现，会在 EngineSelector 上层加一个 CoordinatorAgent，它负责把大任务拆成小 Task 分配给不同的 ExpertAgent（代码 Agent、搜索 Agent、文件 Agent），最后汇总结果。"

### 题目 4：多 Agent 系统中如何避免重复执行同一任务？如何保证最终一致性？幂等怎么设计？

**推荐回答：**

> "SpringClaw 目前是单 Agent，但幂等问题在自主循环中同样存在——模型可能在同一轮循环中对同一文件写两次。
>
> 我做了三层防护：1) 请求级互斥——每个请求有唯一 requestId，Redis SETNX 做会话锁（`springclaw:guard:lock:{sessionKey}`），保证同一会话不会并发执行两个请求；2) 工具级跟踪——AutonomousExecutionTracker 用 CopyOnWriteArrayList 记录每次工具调用的目标文件和参数，如果同一文件被写两次，只有最后一次生效；3) 风险门控——write 级别以上的操作需要用户确认（AgentActionProposalService），用户不确认就不执行。
>
> 最终一致性方面：自主循环是有界循环（最多 maxAutonomousSteps 步），每步是串行执行的，不存在并发写冲突。如果要做分布式幂等，我会用 requestId + Redis 去重表：执行前检查 `SETNX idempotent:{requestId}`，已存在则跳过。"

**代码映射：** `RedisChatGuardService.java` lines 103-162（分布式锁），`AutonomousExecutionTracker.java`（工具调用跟踪），`AgentActionProposalService.java`（风险门控）

### 题目 5：Harness Engineering 是什么？Agent Harness 层一般包含哪些职责？

**推荐回答：**

> "Harness Engineering 是 Agent 系统的'基础设施层'——它不关心 Agent 做什么任务，只关心 Agent 怎么可靠地运行。就像赛车的 harness（安全带），不管车手怎么开，harness 保证他不出事。
>
> Agent Harness 层一般包含五个职责，SpringClaw 中都有对应实现：
>
> 1) **调度**——决定哪个请求在什么时候执行。SpringClaw 用 Redis 分布式锁做会话级互斥，XXL-Job 做定时任务调度。
>
> 2) **超时/熔断**——模型调用超时或连续失败时自动降级。SpringClaw 的 ModelTransportGuardService 实现了完整的熔断器：每个 Provider 跟踪连续失败次数，超过阈值（默认 2 次）进入 90 秒冷却期。冷却期间自动切换到其他 Provider。
>
> 3) **上下文注入**——在模型调用前组装上下文。ContextAssembler 从短期记忆（最近 8 轮对话）+ 长期语义记忆（Redis 向量存储 top-8 召回）中组装上下文，注入到 system prompt。
>
> 4) **可观测性**——看到 Agent 内部发生了什么。AgentRunTraceService 记录完整的执行轨迹到三张表（agent_run、agent_run_step、tool_invocation），支持 replayRun() 完整回放。Token 使用量精确到 prompt cache hit/miss 级别。
>
> 5) **错误隔离**——一个组件挂掉不影响整体。所有 Redis 操作都有本地 ConcurrentHashMap 降级，模型 Provider 挂了自动 failover 到备用 Provider，RabbitMQ 有死信队列兜底。"

**代码映射：** `ModelTransportGuardService.java`（熔断），`ModelCallExecutor.java`（重试/failover），`ContextAssembler.java`（上下文注入），`AgentRunTraceService.java`（可观测性），`RedisChatGuardService.java`（错误隔离）

### 题目 6：ReAct 范式（Thought-Action-Observation）和 Plan-and-Execute 的区别及适用场景

**推荐回答：**

> "ReAct 和 Plan-and-Execute 是两种不同的 Agent 执行范式。
>
> ReAct 是'边想边做'——每一步模型先 Thought（我要干什么），再 Action（调用工具），再 Observation（看工具返回了什么），然后决定下一步。它适合开放式任务，模型可以根据中间结果灵活调整策略。SpringClaw 的 AutonomousLoopEngine 就是一个 ReAct 变体——模型每步看到工具列表和历史，自己决定调什么工具，工具结果在下一步的 prompt 中可见，循环直到 TASK_COMPLETE。
>
> Plan-and-Execute 是'先想后做'——模型先制定完整计划（步骤 1、2、3），然后按计划执行，执行完可以反思计划是否需要调整。它适合结构化任务，计划先行可以减少错误。SpringClaw 的 OparLoopEngine（Observe-Plan-Act-Reflect）就是 Plan-and-Execute——先让模型生成 PlanResult（包含步骤列表和工具提示），然后逐步执行，执行后反思证据是否充分。
>
> 区别在于：ReAct 每步都重新决策（灵活但可能迷失方向），Plan-and-Execute 先规划再执行（有方向感但计划可能不准确）。SpringClaw 的选择策略是：write/side_effect/dangerous 任务走 AutonomousLoopEngine（ReAct，因为写操作需要根据中间结果灵活调整），read/analysis 任务走 OparLoopEngine（Plan-and-Execute，因为分析任务结构化程度高）。"

**代码映射：** `AutonomousLoopEngine.java` lines 163-373（ReAct 循环），`OparLoopEngine.java`（Plan-and-Execute 循环）

### 题目 7：LangGraph 中 State / Node / Edge / Conditional Edge 分别解决什么问题？Checkpointers 是干嘛的？

**推荐回答：**

> "LangGraph 把 Agent 工作流建模为有向图。核心组件：
>
> **State** 是图中流动的全局状态对象，每个 Node 都可以读写它。比如一个客服 Agent 的 State 可能包含 `{user_query, intent, retrieved_docs, response}`。
>
> **Node** 是执行单元，每个 Node 接收 State、做一些处理、返回更新后的 State。比如 `retrieval_node` 做 RAG 检索，`generation_node` 做答案生成。
>
> **Edge** 是 Node 之间的固定连接——A 执行完一定去 B。
>
> **Conditional Edge** 是动态路由——根据 State 中的某个字段决定下一步去哪个 Node。比如 `if state.intent == 'complaint' → complaint_handler, else → general_handler`。这解决了'不同意图走不同处理流程'的问题。
>
> **Checkpointers** 做状态持久化——在每个 Node 执行后把完整 State 序列化到数据库（SQLite/Postgres）。它支持两个场景：1) Human-in-the-loop——执行到某个 Node 暂停，等用户确认后从 Checkpoint 恢复继续执行；2) 容错——如果 Agent 中途崩溃，可以从最近的 Checkpoint 恢复而不是从头开始。
>
> SpringClaw 没有用 LangGraph，但实现了类似的概念：EngineSelector 对应 Conditional Edge（按条件选引擎），ChatRoutingStateService 对应简化版 Checkpoint（把执行模式存到 Redis），Plan/Act 审批对应 Human-in-the-loop（计划确认后才执行）。"

---

### 题目 8：LangGraph 中怎么实现 Human-in-the-loop（中断恢复 checkpoint）？序列化状态存在哪？

**推荐回答：**

> "LangGraph 的 HITL 通过 interrupt_before / interrupt_after 配置实现。在图的定义中指定 `interrupt_before=['sensitive_tool_node']`，Agent 执行到 sensitive_tool_node 前会自动暂停，把当前 State 通过 Checkpointer 序列化到数据库。前端展示计划内容，用户确认后调用 `graph.invoke(None, config)` 从 Checkpoint 恢复继续执行。
>
> 序列化状态存在 Checkpointer 配置的后端——MemorySaver（内存，开发用）、SqliteSaver（SQLite，单机用）、PostgresSaver（生产用）。序列化内容包括：当前 Node 名称、完整 State 字典、消息历史、工具调用记录。
>
> SpringClaw 的 HITL 实现更简单：AgentActionProposalService 对 write/side_effect/dangerous 操作生成 ActionProposal，前端展示确认对话框，用户确认后才执行工具调用。没有 LangGraph 那么完整的 Checkpoint 机制——我的计划确认是在执行前，不需要中断后恢复。如果要做中断恢复，我会把当前执行进度（已完成的步骤、待执行的步骤）存到 Redis，超时后清理。"

**代码映射：** `AgentActionProposalService.java`（确认提案），`ToolRiskPolicyService.java`（风险分类）

## 第三章：意图识别与多轮对话

### 题目 9：意图识别在多轮对话中怎么维护上下文？怎么处理意图切换（比如中途改需求说"算了我要找日料"）？

**推荐回答：**

> "SpringClaw 做了两层意图维护：
>
> 第一层是每轮独立分类。每次用户发消息，AgentDecisionService 都会重新做一次意图分类——先用规则路由（关键词匹配），如果置信度不够再调模型做语义分类。这意味着用户的每一句话都会被独立理解，不依赖上一轮的意图。
>
> 第二层是意图状态跟踪。每次分类后，系统把当前 intent 和上一轮的 lastIntent 对比。如果意图类型变了（比如从 workspace_write 变成 web_search），标记为 significant change，自动中止当前的 OPAR 计划，清空步骤状态，重新走分类流程。如果意图类型不变但参数变了（比如从'搜索 AI 新闻'变成'搜索 Java 新闻'），保持执行模式不变。
>
> 对于'算了我要找日料'这种场景：前一个意图可能是某个工具调用（比如代码分析），新意图是完全不同的方向（web_search 或 local_files）。系统会检测到 significant change，中止当前计划，重新分类新意图，然后选择合适的引擎执行。用户不需要显式'取消'——系统自动感知意图跳变。
>
> 意图状态存在 Redis（`springclaw:chat:routing:mode`），带 TTL，服务重启后仍然保留。"

**代码映射：** `AgentDecisionService.java` lines 63-99（混合分类），`ChatRoutingStateService.java`（状态持久化），`ChatRoutingPolicyService.java`（自动升级/降级）

### 题目 10：Slot Filling——用户说"我想找个附近人均50左右好吃的川菜馆"，怎么做意图识别与槽位填充？NER + 分类是微调还是 LLM？

**推荐回答：**

> "这个场景的意图是 local_search（本地生活搜索），需要填充的槽位是：
> - cuisine = 川菜
> - price_range = 人均50
> - location = 附近（需要结合用户定位）
> - quality = 好吃的（主观评价，转化为排序权重）
>
> 实现路线有两种：
>
> **微调 NER + 分类器**：训练一个 BERT 级别的模型做 token 级 NER（提取'川菜''人均50'）和句级分类（识别意图为 local_search）。优点是快（毫秒级）、准（对训练过的领域 F1 > 0.95）。缺点是泛化差——训练数据里没有'日料'就识别不了。
>
> **LLM + Function Calling**：把用户的话直接给模型，prompt 里定义好意图类型和槽位 schema，让模型输出 JSON。优点是泛化强——'日料''法餐'都能处理，不需要重新训练。缺点是慢（秒级）和贵。
>
> SpringClaw 用的是第二种——AgentDecisionService 的模型路由就是这个模式。prompt 约束输出 JSON（'你是 Agent 路由器，只输出 JSON，不要解释'），解析后得到 intent + selectedCapabilities + riskLevel。对于高频明确的意图（比如'打开桌面文件'），规则路由先行，不需要调模型。"

### 题目 11：如果 LLM 做意图识别，怎么设计 System Prompt 保证输出稳定？如何处理模型乱返回格式？

**推荐回答：**

> "SpringClaw 做了四层保障：
>
> 第一层，**prompt 硬约束**。System prompt 明确写'你是 Agent 路由器。只输出 JSON，不要解释。'并且给出完整的 JSON schema 示例。用 `.replace()` 而不是 `String.format()` 来注入变量，避免用户输入中的 `%` 导致格式化异常。
>
> 第二层，**JSON 边界提取**。不直接 `JSON.parse(全文)`，而是找第一个 `{` 到最后一个 `}` 之间的子串。这样即使模型在 JSON 前输出了'好的，这是结果：'这类前缀，也能正确提取。
>
> 第三层，**字段级降级**。解析出的 JSON 如果缺少某个字段（比如没有 riskLevel），不会报错，而是用规则路由的对应字段做 fallback。代码里是 `modelDecision.riskLevel() != null ? modelDecision : new AgentDecision(...)`。
>
> 第四层，**整体降级**。如果 JSON 解析完全失败（模型输出了一堆乱码），catch 异常后直接返回规则路由的结果。同时 log.warn 记录失败原因，方便排查。
>
> 更稳健的做法是用模型提供商的 JSON mode（OpenAI 的 response_format: json_object）或通过 Function Calling 强制结构化输出。SpringClaw 目前没有用这些，因为规则路由已经处理了大部分请求，模型路由只是兜底。"

**代码映射：** `AgentDecisionService.java` lines 168-202（prompt 模板），lines 216-235（JSON 解析 + 降级）

## 第四章：RAG 与记忆

### 题目 12：RAG 检索结果不理想怎么优化——Query Rewriting / HyDE / Multi-Query / Rerank？

**推荐回答：**

> "这四种是 RAG 检索质量的渐进优化手段：
>
> **Query Rewriting** 是最基础的——把用户的原始查询改写成更适合检索的形式。比如用户说'它怎么用'，结合上下文改写为'SpringClaw 的 AutonomousLoopEngine 使用方法'。SpringClaw 在 SemanticMemoryAdvisor 中做了基础版——代词消解和短问题补全。培训助手项目中做了更完整的版本，复杂业务意图识别准确率从 75% 提升到 94%。
>
> **HyDE（Hypothetical Document Embeddings）** 是让模型先生成一个假设性回答，然后用这个回答的 embedding 去检索，而不是用原始问题的 embedding。好处是假设性回答和真实文档在语义空间上更接近。我没有实现，但如果 Query Rewriting 不够用，这是下一步。
>
> **Multi-Query** 是生成多个查询变体（比如'什么是 Agent''Agent 的定义''AI Agent 概念'），分别检索取并集。好处是召回率更高，坏处是多次检索的延迟和成本。
>
> **Rerank（Cross-Encoder）** 是对向量检索返回的 top-K 结果做第二轮精排。向量检索是 Bi-Encoder（query 和 doc 分别编码后算余弦），速度快但精度有限。Cross-Encoder 把 query 和 doc 拼接后联合编码，精度更高但更慢。所以先 Bi-Encoder 粗排 top-20，再 Cross-Encoder 精排 top-5，是性价比最高的组合。
>
> 如果效果不够好，我会优先加 Rerank，因为 ROI 最高——不需要改检索流程，只在返回结果后加一层精排。"

**代码映射：** `SemanticMemoryAdvisor.java`（Naive RAG 实现），`VectorMemoryService.java`（向量检索）

### 题目 13：RAG 全流程踩坑——chunk 大小怎么选？为什么不能用过大或过小的 chunk？

**推荐回答：**

> "chunk 大小是 RAG 中最关键的超参数之一。
>
> **太小的 chunk**（比如 50 token）：语义碎片化，一个完整的概念被拆到多个 chunk 里。检索可能命中了某个 chunk 但它只有半句话，模型无法基于这个碎片生成好的回答。
>
> **太大的 chunk**（比如 2000 token）：引入噪音。一个 chunk 里可能包含多个话题，向量相似度会被无关内容拉低。而且大 chunk 占用更多 context window，减少能注入的 chunk 数量。
>
> 一般建议 256-512 token + 50 token 重叠（overlap）。重叠是为了防止关键信息恰好被切在两个 chunk 的边界上。
>
> 培训助手项目中，我对不同格式做了适配：PDF 按段落切分（保留标题层级），Word 按章节切分，Markdown 按 `##` 标题切分。chunk 大小 384 token，overlap 64 token。
>
> 进阶做法是 parent-child chunk——小 chunk 用于检索（精度高），命中后返回它所在的大 chunk（上下文完整）。"

### 题目 14：向量数据库用的是什么？Milvus / Chroma / Faiss 怎么选？

**推荐回答：**

> "SpringClaw 用的是 Redis Vector Store（Redis Stack + RediSearch），因为 Redis 已经是项目的基础设施（缓存、分布式锁都在用），加一个向量存储不需要额外运维。
>

### 题目 15：Agentic RAG 和 Naive RAG 的区别？什么时候需要 Agentic RAG？

**推荐回答：**

> "Naive RAG 是单次检索：用户提问 → embed → 向量检索 → 注入 prompt → 生成回答。问题是如果第一次检索没命中相关文档，整个回答就废了。
>
> Agentic RAG 在检索和生成之间加了 Agent 行为——迭代、反思、路由：
> 1) **迭代检索**：第一次检索结果不好，Agent 改写 query 重新检索
> 2) **自我反思**：Agent 评估检索到的文档是否相关，不相关就换策略
> 3) **多源路由**：Agent 根据问题类型决定从哪个数据源检索（向量库 / 知识图谱 / Web 搜索）
>
> SpringClaw 两层都有。SemanticMemoryAdvisor 是 Naive RAG——直接 embed query 然后 similarity search。但 AgentRuntimeEngine 实现了 Agentic RAG 的核心循环：执行工具调用（可以包含检索）→ 模型做 evidence reflection（评估结果是否充分）→ 如果不充分生成 nextQuery 重新检索 → 最多 3 轮。反思的 system prompt 明确要求模型判断'sufficient=true only when results can answer the original question'。
>
> 什么时候需要 Agentic RAG：当你的知识源多（文档库 + Web + 数据库）、问题复杂（需要多跳推理）、或对准确率要求高（不能接受检索失败就直接瞎编）。如果只是简单的 FAQ 问答，Naive RAG 够了。"

**代码映射：** `SemanticMemoryAdvisor.java`（Naive RAG），`AgentRuntimeEngine.java` lines 147-271（Agentic RAG 循环 + 反思）

### 题目 16：MCP 协议了解吗？相关的协议标准？

**推荐回答：**

> "MCP（Model Context Protocol）是 Anthropic 提出的标准协议，定义了 LLM 与外部工具/数据源的通信规范。核心概念有三个：
>
> **Resources**：上下文数据源（文件、数据库记录、API 返回）。类似 REST 的 GET——Agent 读取上下文。
>
> **Tools**：可调用的动作（写文件、执行命令、发请求）。类似 REST 的 POST——Agent 执行操作。
>
> **Prompts**：预定义的 prompt 模板，可以被 Agent 按需调用。
>
> MCP 的价值在于标准化——不管底层是什么工具，只要实现了 MCP Server 接口，任何 Agent 都能调用。类似 USB 统一了外设接口。
>
> SpringClaw 目前用的是 Spring AI 原生的 Function Calling（@Tool 注解），不是 MCP。如果要做 MCP 适配，我会在 ToolOrchestrator 层加一个 MCP Server 适配器——把现有的 13 个 ToolPack 暴露为 MCP Tools，同时支持连接外部 MCP Server 扩展工具集。"

### 题目 17：Function Calling / Tool Calling 的实现机制？

**推荐回答：**

> "Spring AI 的 Function Calling 机制是这样的：
>
> 1) **声明**：用 @Tool 注解标记方法，Spring AI 自动提取方法签名（名称、参数类型、JavaDoc 描述）生成 JSON Schema，注册到模型的 tools 参数中。

## 第六章：记忆管理

### 题目 18：上下文如何管理？长期记忆、短期记忆、工作记忆如何划分？

**推荐回答：**

> "SpringClaw 实现了三层记忆：
>
> **短期记忆**（Short-term）：滑动窗口，保留最近 8 轮对话。存储在 MySQL 的 message_event 表中，按 sessionKey 查询。用于维持当前对话的上下文连续性——比如用户说'打开它'时，系统需要知道'它'指的是上一轮提到的哪个文件。

>
> 和人类类比：短期记忆 = 当前对话的工作台，长期记忆 = 你的知识库，工作记忆 = 你手上正在做的事情的临时便签。"

## 第七章：模型与微调

### 题目 19：模型微调了解过 LoRA / QLoRA 吗？

**推荐回答：**

> "了解原理。LoRA（Low-Rank Adaptation）的核心思想是不更新原始权重矩阵 W，而是在旁边加一个低秩分解矩阵 W' = W + AB，其中 A 是 d×r 矩阵，B 是 r×d 矩阵，r << d。训练时只更新 A 和 B，参数量从数十亿降到数百万。推理时 A 和 B 可以合并回 W，没有额外延迟。
>
> QLoRA 在 LoRA 基础上加了 4-bit 量化——原始模型用 NF4 量化加载，LoRA 适配器用 bf16 训练。这样一张 24GB 的消费级 GPU 就能微调 70B 模型。
>
> 但我没有实操经验。我的方向是 Agent 工程化——怎么路由意图、怎么编排工具、怎么保证执行可靠性。微调是模型层的事情，如果要做我会用 Hugging Face PEFT 库。在 SpringClaw 中，如果意图分类的准确率不够，我的第一选择是优化 prompt 和规则路由（工程方案），而不是微调模型（训练方案），因为工程方案迭代更快。"

### 题目 21：Token 计费机制如何理解？如何降低 token 消耗？

**推荐回答：**

> "大模型按 token 计费，分 input token（你发给模型的）和 output token（模型返回的），output 通常比 input 贵 2-4 倍。部分提供商（Anthropic/OpenAI）支持 prompt cache——如果连续请求的 prompt 前缀相同，缓存命中的 token 价格打折（Anthropic 打 1 折）。

> Token 使用量通过 TokenCacheUsageExtractor 精确记录到 prompt cache hit/miss 级别，持久化到 llm_usage_record 表，可以按时间段统计消耗趋势。"

### 题目 24：高并发场景下，如何保证 AI 模型调用的可用性和稳定性？

### 题目 26：语义缓存怎么做？如何减少重复调用？

**推荐回答：**

> "语义缓存的核心思想是：如果新问题和之前某个问题的 embedding 相似度超过阈值，直接返回缓存的回答，不再调模型。
>
> 实现方案：1) 用 embedding 模型把每个 query 编码成向量；2) 存到向量数据库（SpringClaw 可以用 Redis Vector Store 的独立 index `springclaw-cache`）；3) 新 query 来时先查缓存——如果最近邻距离 < 阈值（比如 0.08，对应余弦相似度 > 0.92），返回缓存的 response；4) 缓存条目设 TTL（比如 24 小时），过期自动清理。
>
> SpringClaw 目前没有实现语义缓存，但实现了工具级缓存——天气、汇率、新闻这些外部 API 调用用 Caffeine 做 TTL 缓存。如果加语义缓存，关键挑战是阈值选择——太低会导致不相关的问题命中缓存（false positive），太高则缓存命中率太低。可以做一个 admin 接口让运维动态调整阈值。"

### 题目 28：MySQL 事务四大特性 ACID？MVCC 怎么实现的？

**推荐回答：**

> "**ACID：**
> - Atomicity（原子性）：事务要么全做要么全不做，通过 undo log 实现回滚。
> - Consistency（一致性）：事务前后数据库从一个一致状态到另一个一致状态，由其他三个特性共同保证。
> - Isolation（隔离性）：并发事务互不干扰，通过锁 + MVCC 实现。
> - Durability（持久性）：事务提交后数据不丢，通过 redo log 实现。
>
> **MVCC（Multi-Version Concurrency Control）：**
> InnoDB 每行数据有两个隐藏列：DB_TRX_ID（最后修改的事务 ID）和 DB_ROLL_PTR（指向 undo log 的指针）。undo log 形成版本链——当前版本 → 上一版本 → 更早版本。
>
> 读操作根据隔离级别决定看哪个版本：
> - Read Committed：只看已提交事务的最新版本（每次 SELECT 生成新快照）
> - Repeatable Read：看事务开始时的快照（整个事务期间用同一个快照，解决不可重复读）
>
> 写操作走悲观锁（SELECT ... FOR UPDATE / 排他锁），读操作走 MVCC 快照读，读写不冲突。"

### 题目 29：关于并发，异步调用怎么做？

**推荐回答：**

> "SpringClaw 有三种异步模式：
>
> 1) **CompletableFuture 并行调用**：虚贝云游项目中，商品详情接口需要同时查商品、价格、库存、详情四个远程服务。用 CompletableFuture.supplyAsync() + 自定义线程池并行执行，allOf() 等待全部完成，RT 从 450ms 降到 85ms。
>
> 2) **SSE 流式输出**：ChatServiceImpl 通过 SseEmitter 实现 Server-Sent Events，模型生成的 token 实时推送到前端。引擎自己管理 SSE 生命周期——成功时自行完成 emitter，失败时委托给 fallbackHandler 降级。
>
> 3) **RabbitMQ 消息队列**：聊天请求可以通过 MQ 异步投递，实现流量削峰。消费端处理失败时消息进入死信队列，不会丢失。
>
> 线程安全方面，AutonomousExecutionTracker 用 CopyOnWriteArrayList 和 ConcurrentHashMap 做防御性设计（为未来异步执行做准备），Redis 锁保证会话级互斥。"

## 第十章：框架对比

### 题目 30：LangChain 和 LangGraph 的核心区别？为什么 LangChain 更适合多步 ReAct Agent？

**推荐回答：**

> "严格来说，LangGraph 是 LangChain 团队做的，不是竞争关系而是互补。
>
> **LangChain** 是链式调用框架——核心抽象是 Chain（把多个步骤串起来）和 Agent（让模型决定调哪个工具）。它的 ReAct Agent 是线性的：Thought → Action → Observation → Thought → ... → Final Answer。适合单 Agent、线性流程。
>
> **LangGraph** 是图编排框架——核心抽象是 State + Node + Edge。它把 Agent 工作流建模为有向图，支持循环（一个 Node 执行完可以回到之前的 Node）、分支（Conditional Edge）、并行和 Human-in-the-loop。适合多 Agent、复杂流程。
>
> 'LangChain 更适合多步 ReAct Agent'这个说法不太准确——LangChain 的 create_react_agent() 开箱即用，代码量少，适合快速搭建一个简单的 ReAct Agent。LangGraph 需要手动定义图结构，代码量更多但控制力更强。如果只是'给一个问题让 Agent 自己调工具解决'，LangChain 够了；如果需要'Agent A 分析完后让 Agent B 写代码再让 Agent C 审查'，LangGraph 更合适。"

### 题目 31：关于多 Agent 框架 CrewAI 了解吗？

**推荐回答：**

> "了解核心概念但没在生产中使用过。CrewAI 的设计哲学是'像组建团队一样组建 Agent'：
>
> - **Agent** 有 role（角色）、goal（目标）、backstory（背景故事），类似一个团队成员的人设
> - **Task** 有 description（任务描述）、expected_output（期望输出）、assigned_agent（分配的 Agent）

## 第十一章：成本控制

### 题目 32：多 Agent 系统成本怎么控制？

**推荐回答：**

> "多 Agent 系统的成本主要来自模型调用次数——每多一个 Agent 就多一次 LLM 调用。SpringClaw 做了三层控制：
>
> 1) **语义缓存**（规划中）：相似问题命中缓存直接返回，避免重复调模型。用 Redis Vector Store 的独立 index 实现，相似度阈值 0.92。
>
> 2) **工具级缓存**（已实现）：天气、汇率、新闻等外部 API 用 Caffeine 做 TTL 缓存（5-30 分钟），短期内重复查询不调 API 也不调模型。
>
> 3) **确定性优先**（已实现）：能用规则回答的问题不走模型。OparLoopEngine 的 control-plane 对'你是什么模型''列出所有模型'直接返回确定性答案。AgentQualityEvaluator 的 costScore 对超过 3 次工具调用或 15 秒以上的请求扣分，倒逼引擎选择更轻的路径。
>
> 还没做的：per-user spending limit（每个用户每月最多花多少钱）、pre-flight cost estimation（执行前估算这次调用大概花多少钱）、prompt compression（缩短 prompt 减少 input token）。"

## 第十二章：可观测性

### 题目 33：Trace / Log 如何设计？

**推荐回答：**

> "SpringClaw 的 AgentRunTraceService 设计了三层 trace 结构：
>
> **agent_run** 表：一次完整的用户请求。记录 requestId、sessionKey、选中的引擎、最终质量分数、总耗时。

## 第十三章：Go 语言基础

### 题目 34：Go 的 channel 有无缓冲区别？close 之后发送会 panic 吗？如何优雅退出 goroutine？

**推荐回答：**

> "**无缓冲 channel**（`make(chan int)`）：发送和接收必须同时就绪，否则阻塞。类似同步通信——发送方等到接收方拿走数据才继续。
>
> **有缓冲 channel**（`make(chan int, 10)`）：缓冲区没满时发送不阻塞，缓冲区空了时接收阻塞。类似异步队列。
>
> **close 之后发送会 panic**——`send on closed channel` 是 runtime panic。但 close 之后接收不会 panic，会返回零值（如果是带 ok 的接收 `v, ok := <-ch`，ok 为 false）。
>
> **优雅退出 goroutine** 用 `context.WithCancel` + `select`：
> ```go
> ctx, cancel := context.WithCancel(context.Background())
> go func() {
>     for {
>         select {
>         case <-ctx.Done():
>             return // 收到取消信号，退出
>         case data := <-ch:
>             process(data)
>         }
>     }
> }()
> // 需要退出时
> cancel()
> ```
> 不要用 close(ch) 来通知退出——如果有多个 goroutine 往同一个 channel 发送，close 会导致 panic。context 是更安全的选择。"

## 第十四章：手撕算法

### 题目 35：LeetCode——找出字符串中最长公共前缀

**推荐回答：**

> "横向扫描法：取第一个字符串作为初始前缀，逐个和后面的字符串比较，每次比较都缩短前缀到公共部分。如果前缀变成空串就提前返回。
>
> ```java
> public String longestCommonPrefix(String[] strs) {
>     if (strs == null || strs.length == 0) return "";
>     String prefix = strs[0];
>     for (int i = 1; i < strs.length; i++) {
>         while (!strs[i].startsWith(prefix)) {
>             prefix = prefix.substring(0, prefix.length() - 1);
>             if (prefix.isEmpty()) return "";
>         }
>     }
>     return prefix;
> }
> ```
> 时间 O(S) 其中 S 是所有字符串的字符总数，空间 O(1)。"
