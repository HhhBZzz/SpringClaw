# RESUME_PROJECT_VERSIONS

下面给出两套可直接用于校招简历的项目文案：

- `单项目增强版`：只保留第一个项目，把你能讲稳的多渠道接入、会话隔离、记忆治理能力融合进去。
- `双项目版`：第一个项目继续放第一，第二个项目作为工程化补强。

默认建议优先使用 `单项目增强版`，因为信息更集中，也更适合 AI 应用 / 智能体开发岗位的一页式简历。

## 单项目增强版

### 项目名称

企业智能知识助手平台（多渠道接入与多轮会话）

### 技术栈

Python / FastAPI / LangChain / RAGFlow / MongoDB / SSE / Qwen API

### 软件架构

前后端分离的智能问答后端架构，整体分为 `消息接入层`、`服务编排层`、`意图识别与查询增强层`、`知识检索层`、`会话记忆层` 五部分；其中 API 请求统一进入 FastAPI 服务，并预留企业消息平台的渠道适配扩展点，用于承接飞书、企业微信等场景下的统一消息映射；后续由 LangChain 编排意图识别、Query Rewrite、RAGFlow 检索和大模型生成链路，并通过 MongoDB 完成会话持久化与上下文复用。

### 开发工具

Git / Docker / Swagger(OpenAPI) / curl

### 项目背景

面向企业内部知识查询场景，围绕“新员工入职培训资料查询、请假报销制度问答、采购流程说明、部门文档检索”等典型业务需求，解决 PDF、Word、Markdown 等多格式资料分散、传统关键词检索效率低、人工答疑成本高的问题；项目以 Web/API 问答后端为主体，并兼顾企业消息平台接入场景下的统一消息映射与上下文复用需求，提升企业内部知识查询与使用效率。

### 个人职责

1. 面向入职培训、请假报销、采购流程、部门资料检索等 `4` 类企业内部知识场景，负责搭建 `意图识别 -> Query Rewrite -> 知识检索 -> 答案生成 -> 会话落库` 的 `5` 段式核心问答链路，支持知识库问答与通用聊天 `2` 类场景路由。
2. 基于 Function Calling 实现用户意图识别与助手路由，打通“知识库问答 / 通用聊天” `2` 类问答模式，并支持“用户指定知识助手回答”与“系统自动选择最优助手回答” `2` 种路由方式。
3. 对接 RAGFlow 知识助手能力，结合历史对话进行 Query Rewrite，补全代词、省略信息和上下文，提升多轮追问场景下的检索命中效果与回答连贯性。
4. 面向企业消息平台场景设计统一消息接入与渠道适配方案，抽象 `session_id / user_id` 会话映射规则，使系统在后续接入飞书、企业微信等平台时可复用同一套多轮问答与会话记忆链路。
5. 基于 FastAPI 提供非流式问答、SSE 流式问答、会话记忆清理 `3` 类核心接口能力，并设计 MongoDB 会话记忆持久化方案，在依赖异常场景下回退至内存记忆，保证多轮会话能力持续可用。

### 技术亮点

1. 基于 Function Calling 实现“知识库问答 / 通用聊天 / 助手选择”动态路由，避免固定流程只适配单一输入类型的问题，更适合企业内部多类型问答场景。
2. 增加 Query Rewrite 能力，对“还有吗”“它是什么”“他是谁”类追问做上下文补全，形成从单轮问答到多轮连续追问的链路闭环。
3. 通过 `session_id` 维度做会话隔离，并结合 MongoDB 持久化与内存降级，兼顾上下文连续性、异常可恢复性和服务可用性。
4. 将消息入口与核心 RAG 问答链路解耦，抽象统一消息模型与会话映射规则，使系统在不改动核心检索与生成流程的前提下具备扩展到飞书、企业微信等平台的能力，降低多渠道接入下的维护成本。
5. 在项目落地过程中补齐了 RAG 应用的关键工程化能力，包括流式输出、记忆治理、异常降级和多入口接入，对 AI 应用从“能回答”到“能稳定服务”的实现路径形成了完整认知。

### 面试表达口径

- 企业消息平台能力只表述为：`围绕企业消息入口设计了统一消息映射与会话复用方案。`
- 不写成“已完整上线多平台接入”，不主动引申到 SDK 细节、复杂群聊路由、多租户或权限后台。
- `MCP` 统一替换为：`基于 Function Calling 的意图识别与助手路由。`

## 双项目版

### 项目一

#### 项目名称

企业智能知识助手平台（多渠道接入与多轮会话）

#### 技术栈

Python / FastAPI / LangChain / RAGFlow / MongoDB / SSE / Qwen API

#### 软件架构

前后端分离的智能问答后端架构，核心由消息接入层、问答编排层、检索增强层和会话记忆层组成；通过统一入口承接 API 与企业 IM 平台消息，当前完成飞书接入，并可复用同一套 RAG 问答链路扩展至企业微信等平台。

#### 开发工具

Git / Docker / Swagger(OpenAPI) / curl

#### 项目背景

面向企业制度、培训资料、业务文档查询场景，围绕“新员工入职培训、请假报销制度查询、采购流程说明”等典型企业内部服务需求，构建支持多轮追问与知识检索的智能问答后端，解决企业资料分散、问答依赖人工、文档格式不统一带来的查询效率问题。

#### 个人职责

1. 负责搭建 `意图识别 -> Query Rewrite -> RAG 检索 -> 答案生成 -> 会话记忆持久化` 主链路，支持知识问答与通用聊天的动态路由。
2. 实现知识助手自动选择与指定助手回答能力，结合历史对话完成代词补全与查询增强，提升多轮追问场景下的检索准确性。
3. 设计统一消息入口与渠道适配逻辑，当前完成 API 与飞书接入，统一用户会话标识与上下文复用规则，并支持后续扩展至企业微信等平台。
4. 实现 SSE 流式返回、会话记忆清理与 MongoDB 持久化/内存降级机制，保障服务在多轮场景下的稳定运行。

### 项目二

#### 项目名称

SpringClaw 企业 AI Agent 平台

#### 技术栈

Java / Spring Boot / Spring AI / MySQL / Redis / RabbitMQ / Feishu

#### 软件架构

分层式企业 AI Agent 后端架构，包含控制器接入层、Agent 服务编排层、模型运行时层、记忆存储层和渠道接入层；通过 Spring AI 组织模型调用与工具执行，通过 MySQL 与 Redis 承担短期事件留痕和长期语义记忆。

#### 开发工具

Git / Maven / Docker / IntelliJ IDEA

#### 项目背景

面向企业内部智能助手场景，建设具备多模型运行时、双轨记忆、渠道接入与基础治理能力的 AI Agent 后端，支持 API 与企业消息平台接入，可用于企业内部问答、通知触达、基础运营查询等场景，并为后续扩展工具调用和运营管理提供基础设施。

#### 个人职责

1. 参与设计多模型运行时，支持不同 provider/model 的切换与失败降级，提升模型调用链路的灵活性与可用性。
2. 实现 MySQL 事件流与 Redis 向量检索结合的双轨记忆方案，兼顾会话留痕审计与长期语义召回。
3. 补齐飞书接入、消息留痕与基础运营能力，为企业内部多渠道智能助手接入提供可持续演进的后端基础。

## 使用建议

- `单项目增强版` 更适合校招一页简历，重点集中，面试也更容易讲稳。
- `双项目版` 更适合你想同时体现 `RAG 应用能力` 和 `平台工程化能力` 的场景。
- 如果投递岗位明显偏 `AI 应用 / 智能体开发`，建议优先使用 `单项目增强版`。
- 如果投递岗位同时看重 `Java 后端基础` 和 `AI 落地能力`，可以使用 `双项目版`。
- 如果需要写量化，优先使用“场景数、链路段数、接口数、入口数、模式数”这类结构化数字，不建议编造时延、准确率、日活或调用量。
